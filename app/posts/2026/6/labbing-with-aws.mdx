I've been doing some research for a new project, and my good friend suggested that I add AWS to my skillset. I've done maybe 20 hours of AWS certification coursework in the past, and I've never built anything other than an EC2 with amazon linux to run a minecraft server back in 2021.

So, I've had to research some project ideas that would familiarize myself with AWS, its foundational products, and how they all integrate with one another.

I've come up with one or two interesting concepts, but neither really seem so interesting for me to try and plan and complete. 

First, my friend and I thought about utilizing some tools that might be interesting, like building an API that stores secrets in the secrets manager, and somehow integrating that in the cloud.

But something else came across my mind, because it's relevant in my current job. What about AWS hosted backups?

Sold! In this blog post, we're going to be building an automated backup system fully in the cloud, with Lambdas, S3 buckets, and email alerts!

###  Planning

I began researching what the scope for a project like this might look like. I pretty much immediately ran into one of the core principles of AWS: infinite scalability. I mean think about it, I'm coming from a homelab where I have hand-me-down machines running on 8+ year old hardware.

It became immediately apparent that I needed to plan this entire project out before I touched anything. I understood that the infinitely scaling resource pools also meant that any project, both prod and dev, also had infinite scope. Exciting, sure, but you have to be kind of responsible and purposeful with AWS.

Anyways, the architecture will look as follows: I'll create an S3 bucket that is only available to users and objects within my AWS tenant with a 30 day object lifecycle, I'll need a Lambda to trigger when an object is put into the bucket, and an SNS node to email me the status of the event, whether it was successful or not.

Sounds easy enough! I mean, I handle an actual production M365 environment at work including things like Intune, Entra, Exchange, and Defender. Can't be that hard. Let's get started.

### ACLs, Object locking, Versioning, and encryption.

The language about having no ACLs makes it sound publicly accessible and open to the internet, but that's not the case. It's actually the opposite. Let's liken this setting to a Python if statement.

```python
public = False

if public:
    # allow access
    pass
```

Makes sense now right? If your S3 bucket is private, why would you need ACLs to control access to it? So skip it entirely. If it's public, we apply controls. Simple as.

Any good backup system needs versioning. I don't want backups of just what was on my dataset only the last time I ran it, I want backups from 5, 10, 30 days prior, so we enable this.

SSE-S3 encryption is standard symmetric key encryption (AES-256), that's really not special.

Object locking is interesting. While for my purposes it's actually kind of annoying, what it does is lock down new data put into the bucket for X amount of time. So this could be 1 day, 30 days, 180 days, etc. It follows a concept called WORM or write once, read many. I'm turning it on to say I used it.

![S3 object lock configuration showing governance mode with a 7-day retention period](/posts/labbing-with-aws/s3-object-lock.png)

Next I'm going to make use of object locking set retention from infinite to 7 days with governance mode, which just means I can come in here with an elevated account and clear the bucket out.

![S3 lifecycle management rule configured to expire objects after 30 days](/posts/labbing-with-aws/data-lifecycle.png)

And then I'll setup lifecycle management here. I never explained the reasoning for this setting, but the idea is that you want backups, though you don't necessarily want infinite backups. If these were like virtual machine backups, you'd be spending a lot of money maintaining these buckets. So lifecycle helps manage that.
