This is the infrastructure I learn on. Four Proxmox nodes built from decommissioned
HP EliteDesk SFF machines and a ThinkPad, a gigabit network with real VLAN
segmentation, and an automation layer that means I stopped clicking through UIs
to provision things. None of it is glamorous. That's the point.

## Network

The edge is a NanoPi R5S running OpenWRT — cheap, fanless, and more than capable
of routing a home network. A Cisco Business 220 is the core switch; a small
TP-Link managed switch handles the cabling where things get awkward. Wireless is
a TP-Link Deco mesh with the main node wired back to the Cisco.

The network is still all gigabit. I'm not in a hurry to upgrade — the current goal
is clean segmentation and enough visibility to understand what broke before blaming
the wrong layer.

## Cluster

PVE1 is the ThinkPad; PVE2 through PVE4 are the EliteDesk machines. One of them
carries a Quadro P400 that handles Jellyfin transcoding. DNS is a three-hop
chain: AdGuard takes all queries first and drops denylisted domains, Technitium
handles the authoritative internal zone (lan.hwasani.com), and anything not in
that zone goes to Unbound for recursive resolution. Nginx Proxy Manager
terminates TLS for internal services. Headscale handles remote access to the
Proxmox management interface.

## Automation and Storage

Gitea holds the configuration. Terraform provisions VMs against the Proxmox API.
Semaphore runs the Ansible jobs. Baseline config lands without touching every
system by hand, which is mostly the point — the lab should feel like engineering
practice, not a pile of machines I SSH into individually.

Storage is Ceph after I moved away from local-only. I wrote about that migration
and the failure that forced it in the [storage disaster recovery post](/storage-disaster-recovery).
Next thing I want to try is connecting the RGW bucket to Cloudflare and testing
how CF One mesh patterns can expose object storage to Workers.
